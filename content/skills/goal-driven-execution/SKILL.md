# Goal-Driven Execution

검증 가능한 목표로 변환하라.

작업을 검증 가능한 목표로 변환한 후 수행한다:

- "검증 추가" → "잘못된 입력에 대한 테스트를 작성하고 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
- "X 리팩토링" → "변경 전후 동일한 동작이 유지되는지 확인하기"

## 다단계 작업의 계획 형식

작업이 여러 단계로 나뉘면 시작 전에 다음 형식으로 계획을 제시:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

각 단계가 끝나면 검증 결과를 보고하고 다음 단계로.

## 약한 성공 기준의 함정

"동작하게 해줘", "고쳐줘" 같은 약한 기준은 다음 문제를 만든다:

- 어디까지가 완료인지 불명확 → 끊임없는 추가 질문
- 사용자의 머릿속 기준과 어긋남 → 재작업
- 회귀 발생 시 발견 지연 → 디버깅 어려움

명확한 성공 기준은 작업을 독립적으로 완료 가능하게 한다.

## 검증 방법의 종류

상황에 따라 다음 중 적절한 것을 선택:

- **자동 테스트**: 가장 강력. 가능하면 우선 적용.
- **수동 테스트 시나리오**: 자동화 어려운 UI/통합 작업.
- **빌드/컴파일 통과**: 최소한의 검증.
- **시각적 확인**: UI 변경.
- **로그/메트릭 확인**: 성능, 동작 검증.

검증 방법이 없는 단계는 단계가 아니라 변경 사항일 뿐이다. 게이트를 정의할 수 없으면 작업 단위가 너무 모호한 것.
