# /walkthrough — 코드 투어 (comprehension debt 상환)

AI가 짠 코드의 **구조를 사용자 머리에 남기는** 주문형 투어. 리뷰(결함 찾기)가 아니라
**이해 전이**가 목적이다 — 정거장을 따라 걷고, 끝에 능동 인출(recall)로 마무리한다.

> ①구조 브리핑(RESULT/완료 보고에 자동 포함)이 푸시라면 이 skill은 풀이다 — 브리핑을
> 놓쳤거나, 과거 작업을 다시 익히거나, 특정 시스템을 깊게 걷고 싶을 때 부른다.

## Phase 1 — 범위 결정

인자가 없으면: `git status` + `git diff HEAD` (작업트리 변경), 비어 있으면 `git log --stat -5`를
보여주고 어느 커밋 범위를 걸을지 확인한다. 인자가 있으면 그대로: 커밋범위(`HEAD~3..`),
디렉터리/파일 경로, 또는 시스템명(Grep으로 해당 파일군 식별).

범위가 크면(대략 10파일+) 한 번에 걷지 말고 **계층 투어를 제안**한다 — 1차: 모듈 지도만,
2차: 사용자가 고른 모듈 내부.

## Phase 2 — 투어 순서 설계

파일을 diff 순서가 아니라 **이해 순서**로 정렬한다:

1. **진입점부터** — 외부에서 불리는 곳(공개 API, 이벤트 핸들러, 게임이면 tick/input 경로)
2. **데이터가 흐르는 방향대로** — 생산자 → 변환 → 소비자
3. 타입/데이터 정의는 처음 등장하는 정거장에서 함께
4. 테스트·설정 파일은 마지막 (본문 이해 후 검증 관점으로)

## Phase 3 — 정거장별 안내

정거장(파일 또는 함수군)마다 4가지를 말한다 — 코드 전문을 붙여넣지 말고 **핵심 라인만** 인용:

- **역할**: 이 파일/클래스가 시스템에서 맡은 책임 한 줄
- **연결**: 누가 이걸 부르고, 이건 누굴 부르나 (`파일:라인` 링크로)
- **설계 의도**: 왜 이 모양인가 — 선택된 패턴, 버린 대안, 제약(성능·호환성)
- **주목 지점**: 나중에 고칠 때 건드리게 될 급소 1~2곳

정거장 사이에 사용자가 질문하면 그 자리에서 깊게 들어간다 — 투어 완주보다 이해가 우선.

## Phase 4 — 지도 + 능동 인출

마지막에:

1. **구조 지도** — 전체 흐름을 텍스트 다이어그램 한 장으로 (`A --이벤트--> B --데이터--> C`)
2. **직접 열어볼 파일 3개** — "여기부터 읽으면 전체가 보인다" 순서 포인터
3. **확인 질문 2~3개** — 사용자가 답하게 한다 (예: "X 데이터는 어디서 검증되나요?",
   "Y를 바꾸면 어디가 깨질까요?"). 읽기만 한 지식은 안 남는다 — 인출이 각인이다.
   답을 들으면 맞음/보완을 짚어준다. 사용자가 생략을 원하면 강요하지 않는다.

## 원칙

- vendor-neutral — `git` + Read/Grep만 사용 (Claude·Codex 어디서든 동작)
- 투어는 읽기 전용이다 — 걷다가 발견한 결함은 수정하지 말고 끝에 목록으로만 보고
  (수정은 별도 요청/사이클로)
- 사용자의 기존 이해를 확인하며 걷는다 — 아는 부분은 빠르게, 처음 보는 부분은 천천히
